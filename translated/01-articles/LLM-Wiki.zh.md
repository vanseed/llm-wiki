## LLM Wiki

一种使用大语言模型（LLM）构建个人知识库的模式。

这是一个想法文件，设计目的就是让你复制粘贴给自己的 LLM Agent（例如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。它的目标是传达高层想法，而你的 agent 会与你协作，把具体细节构建出来。

## 核心想法

大多数人与 LLM 和文档打交道的体验，看起来都像检索增强生成（Retrieval-Augmented Generation, RAG）：你上传一组文件，LLM 在查询时检索相关片段，然后生成答案。这种方式可行，但 LLM 每回答一个问题，都在从头重新发现知识。它没有积累。你提出一个需要综合五份文档的细微问题时，LLM 每次都必须找到相关片段并把它们拼接起来。没有任何东西被沉淀下来。NotebookLM、ChatGPT 文件上传，以及大多数 RAG 系统都是这样工作的。

这里的想法不同。LLM 不只是为了查询时从原始文档中检索，而是**增量地构建并维护一个持久 wiki**——一组结构化、相互链接的 markdown 文件，位于你和原始来源之间。当你添加一个新来源时，LLM 不只是把它索引起来以便以后检索。它会阅读该来源，提取关键信息，并将其整合到现有 wiki 中——更新实体页面，修订主题摘要，标注新数据与旧主张相矛盾的地方，强化或挑战不断演化的综合结论。知识只编译一次，然后被*保持为最新*，而不是在每次查询时重新推导。

关键区别在于：**wiki 是一个持久、复利式增长的产物。** 交叉引用已经存在。矛盾已经被标出。综合分析已经反映了你读过的一切。随着你添加的每个来源、提出的每个问题，wiki 会持续变得更加丰富。

你从不（或很少）亲自编写这个 wiki——LLM 会编写并维护它的全部内容。你负责来源筛选、探索，以及提出正确的问题。LLM 负责所有繁重工作——摘要、交叉引用、归档和记账，这些工作会让知识库随着时间推移真正变得有用。实践中，我会把 LLM agent 打开在一侧，把 Obsidian 打开在另一侧。LLM 根据我们的对话进行编辑，而我实时浏览结果——跟随链接、查看图谱视图、阅读更新后的页面。Obsidian 是 IDE；LLM 是程序员；wiki 是代码库。

这可以应用于许多不同语境。举几个例子：

- **个人**：追踪你自己的目标、健康、心理、自我提升——归档日记、文章、播客笔记，并随着时间推移构建出关于你自己的结构化图景。
- **研究**：在数周或数月内深入研究一个主题——阅读论文、文章、报告，并逐步构建一个带有演化论点的综合 wiki。
- **读一本书**：随着阅读进度归档每一章，为人物、主题、情节线索以及它们之间的联系建立页面。到最后，你会拥有一个丰富的伴读 wiki。想想像 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 这样的粉丝 wiki——成千上万相互链接的页面，覆盖人物、地点、事件、语言，由志愿者社区多年构建而成。你可以在个人阅读时构建类似的东西，由 LLM 负责所有交叉引用和维护。
- **商业/团队**：一个由 LLM 维护的内部 wiki，由 Slack 讨论串、会议转录、项目文档、客户电话持续输入。也可以让人类参与审核更新。wiki 会保持最新，因为 LLM 做了团队中没有人愿意做的维护工作。
- **竞品分析、尽职调查、旅行规划、课程笔记、兴趣深挖**——任何你随着时间积累知识，并希望这些知识被组织起来而不是散落各处的场景。

## 架构

有三层：

**原始来源**——你精心筛选的一组来源文档。文章、论文、图片、数据文件。这些是不可变的——LLM 可以读取它们，但绝不修改它们。这是你的事实来源。

**wiki**——一个由 LLM 生成的 markdown 文件目录。摘要、实体页面、概念页面、比较、概览、综合分析。LLM 完全拥有这一层。它创建页面，在新来源到来时更新页面，维护交叉引用，并保持所有内容一致。你阅读它；LLM 编写它。

**schema**——一个文档（例如 Claude Code 的 CLAUDE.md，或 Codex 的 AGENTS.md），告诉 LLM 这个 wiki 如何组织、有哪些约定，以及在摄取来源、回答问题或维护 wiki 时应遵循哪些工作流。这是关键配置文件——它让 LLM 成为一个有纪律的 wiki 维护者，而不是一个通用聊天机器人。随着你们弄清楚什么适合你的领域，你和 LLM 会共同演化这个 schema。

## 操作

**摄取(Ingest)。** 你把一个新来源放入 raw 集合，并告诉 LLM 处理它。一个示例流程是：LLM 读取该来源，与你讨论关键收获，在 wiki 中写入摘要页面，更新索引，更新 wiki 中相关的实体和概念页面，并向日志追加一条记录。单个来源可能会触及 10–15 个 wiki 页面。就我个人而言，我更喜欢一次摄取一个来源并保持参与——我阅读摘要，检查更新，并指导 LLM 应该强调什么。但你也可以在较少监督下批量摄取许多来源。你需要自行发展出适合你风格的工作流，并把它记录在 schema 中，以供未来会话使用。

**查询(Query)。** 你向 wiki 提问。LLM 搜索相关页面，阅读它们，并综合出带引用的答案。根据问题不同，答案可以采用不同形式——markdown 页面、比较表、幻灯片（Marp）、图表（matplotlib）、canvas。重要洞见是：**好的答案可以作为新页面归档回 wiki。** 你要求的比较、某个分析、你发现的某种连接——这些都是有价值的，不应该消失在聊天历史里。这样，你的探索就会像被摄取的来源一样，在知识库中复利增长。

**检查(Lint)。** 定期让 LLM 对 wiki 做健康检查。寻找：页面之间的矛盾、已被较新来源取代的过时主张、没有入链的孤立页面、被提到但尚未拥有自己页面的重要概念、缺失的交叉引用、可通过网络搜索填补的数据空白。LLM 擅长建议新的研究问题和需要寻找的新来源。随着 wiki 增长，这能让它保持健康。

## 索引与日志

两个特殊文件能帮助 LLM（以及你）在 wiki 增长时进行导航。它们服务于不同目的：

**index.md** 面向内容。它是 wiki 中所有内容的目录——每个页面都列出链接、一句话摘要，并可选地包含日期或来源数量等元数据。它按类别组织（实体、概念、来源等）。LLM 会在每次摄取时更新它。回答查询时，LLM 先读取索引以找到相关页面，然后深入阅读这些页面。在中等规模（约 100 个来源、数百个页面）下，这种方式出奇地有效，并且避免了对基于 embedding 的 RAG 基础设施的需求。

**log.md** 面向时间顺序。它是一份仅追加记录，说明什么时候发生了什么——摄取、查询、检查。一个有用的技巧是：如果每条记录都以一致的前缀开头（例如 `## [2026-04-02] ingest | Article Title`），日志就可以用简单的 Unix 工具解析——`grep "^## \\[" log.md | tail -5` 会给出最近 5 条记录。日志提供了 wiki 演化的时间线，并帮助 LLM 理解最近完成了什么。

## 可选：CLI 工具

在某个阶段，你可能会想构建一些小工具，帮助 LLM 更高效地操作 wiki。最显而易见的是一个覆盖 wiki 页面的搜索引擎——在小规模时，索引文件已经足够，但随着 wiki 增长，你会需要真正的搜索。[qmd](https://github.com/tobi/qmd) 是一个不错的选择：它是一个面向 markdown 文件的本地搜索引擎，具备混合 BM25/向量搜索和 LLM 重排序功能，全部在设备本地运行。它既有 CLI（因此 LLM 可以通过 shell 调用），也有 MCP 服务器（因此 LLM 可以把它当作原生工具使用）。你也可以自己构建更简单的东西——当需求出现时，LLM 可以帮助你 vibe-code 一个朴素搜索脚本。

## 技巧与窍门

- **Obsidian Web Clipper** 是一个浏览器扩展，可以把网页文章转换成 markdown。它非常适合快速把来源放入你的 raw 集合。
- **把图片下载到本地。** 在 Obsidian 设置 → 文件与链接中，将“附件文件夹路径”设置为固定目录（例如 `raw/assets/`）。然后在设置 → 快捷键中搜索“Download”，找到“Download attachments for current file”，并把它绑定到一个快捷键（例如 Ctrl+Shift+D）。剪藏文章后按下该快捷键，所有图片都会下载到本地磁盘。这是可选的，但很有用——它让 LLM 能直接查看和引用图片，而不是依赖可能失效的 URL。注意，LLM 不能原生地一次性读取带内联图片的 markdown——变通方法是让 LLM 先读取文本，然后分别查看部分或全部被引用的图片，以获得额外上下文。这有点笨拙，但效果足够好。
- **Obsidian 的图谱视图** 是查看 wiki 形状的最佳方式——什么与什么相连，哪些页面是枢纽，哪些页面是孤岛。
- **Marp** 是一种基于 markdown 的幻灯片格式。Obsidian 有对应插件。它适合直接从 wiki 内容生成演示文稿。
- **Dataview** 是一个 Obsidian 插件，可以基于页面 frontmatter 运行查询。如果你的 LLM 给 wiki 页面添加 YAML frontmatter（标签、日期、来源数量），Dataview 可以生成动态表格和列表。
- wiki 本质上只是一个由 markdown 文件组成的 git 仓库。你免费获得版本历史、分支和协作能力。

## 为什么这有效

维护知识库中乏味的部分不是阅读或思考——而是记账。更新交叉引用、保持摘要为最新、标注新数据何时与旧主张矛盾、在几十个页面之间维持一致性。人类会放弃 wiki，因为维护负担增长得比价值更快。LLM 不会感到无聊，不会忘记更新交叉引用，并且可以在一次处理中触及 15 个文件。wiki 能保持被维护，是因为维护成本接近于零。

人类的工作是筛选来源、引导分析、提出好问题，并思考这一切意味着什么。LLM 的工作则是除此之外的一切。

这个想法在精神上与 Vannevar Bush 的 Memex（1945）有关——一个个人化、经筛选的知识存储，其中带有文档之间的关联路径。Bush 的愿景比后来的互联网更接近这里的想法：私密、主动筛选，并且文档之间的连接与文档本身一样有价值。他无法解决的部分是：谁来维护。LLM 处理了这个问题。

## 注记

本文档有意保持抽象。它描述的是想法，而不是某个具体实现。确切的目录结构、schema 约定、页面格式、工具——所有这些都取决于你的领域、你的偏好和你选择的 LLM。上面提到的一切都是可选且模块化的——选择有用的，忽略无用的。例如：你的来源可能只有文本，所以完全不需要图片处理。你的 wiki 可能足够小，因此索引文件已经满足需求，不需要搜索引擎。你可能不关心幻灯片，只想要 markdown 页面。你可能想要一套完全不同的输出格式。正确的用法是：把它分享给你的 LLM agent，并一起协作实例化一个适合你需求的版本。本文档唯一的任务，是传达这种模式。你的 LLM 可以弄清其余部分。

## 译注

- “wiki is a persistent, compounding artifact” 中的 “compounding” 译为“复利式增长”，强调知识积累会不断叠加增值。
- “bookkeeping” 在本文语境中不是财务记账，而是指维护知识库时的记录、同步、交叉引用和一致性管理。
- “vibe-code” 保留为英文术语，指在 LLM 协助下以较低形式化规划、偏探索式地快速编写代码。
